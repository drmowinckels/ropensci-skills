# Setting up `finchdata` to rOpenSci standards

This walkthrough covers every area that matters for a brand-new R package that wraps an online API: naming, DESCRIPTION, dependencies, documentation, testing, CI, and security. Work through the sections in order — earlier items unblock later ones.

---

## 1. Package name

`finchdata` is all-lowercase, short, and descriptive — a good start. Before you commit to it, verify it is not already on CRAN or Bioconductor and that it is not misleading or offensive:

```r
pak::pkg_name_check("finchdata")
```

Also do a quick web search to confirm the name does not collide with a commercial product's branding.

**Requirement:** the name must not already exist on CRAN or Bioconductor.
**Strong recommendation:** lowercase only, no dots, no unusual characters.

## 3. DESCRIPTION

Fill every field carefully — reviewers read this first.

```dcf
Package: finchdata
Title: Download Bird Observation Data from the <ApiName> API
Version: 0.0.0.9000
Authors@R:
    person("Your", "Name",
           email = "you@example.com",
           role  = c("aut", "cre"),
           comment = c(ORCID = "0000-0000-0000-0000"))
Description: Provides access to bird observation records from the
    <ApiName> database (<https://example-api.org>). Covers retrieval,
    filtering, and light transformation of observation data.
License: MIT + file LICENSE
URL: https://github.com/you/finchdata, https://you.github.io/finchdata
BugReports: https://github.com/you/finchdata/issues
Imports:
    httr2,
    jsonlite,
    cli
Suggests:
    httptest2,
    knitr,
    rmarkdown,
    testthat (>= 3.0.0),
    vcr (>= 1.2.0)
Config/testthat/edition: 3
Encoding: UTF-8
Roxygen: list(markdown = TRUE)
RoxygenNote: 7.x.x
```

Key rules:

- **Title**: Title Case, no trailing period, no "in R" or "with R"; quote the API name with single quotes if it is software/proper name.
- **Description**: do not start with the package name or "This package…"; wrap URLs in `<…>`; for a data-source package you must identify the issuing organisation and link to the data access page.
- **Authors@R**: required form (not the `Author`/`Maintainer` fields); add ORCIDs for all authors.
- **License**: any CRAN- or OSI-accepted license. Mark copyright holders with `cph` role if there are third parties.
- **URL + BugReports**: both required.

## 4. Dependencies

`finchdata` downloads data from an API, so it needs HTTP and JSON support.

| Task                 | Use        | Avoid                    |
| -------------------- | ---------- | ------------------------ |
| HTTP requests        | `httr2`    | `RCurl`, `curl` directly |
| JSON parsing         | `jsonlite` | `rjson`, `RJSONIO`       |
| User-facing messages | `cli`      | `print()`, `cat()`       |

Rules:

- Everything you `::` call goes in `Imports`, not `Depends`.
- Test and vignette tooling (`testthat`, `knitr`, `rmarkdown`, `vcr`, `httptest2`) go in `Suggests`.
- Only add a minimum version constraint (`pkg (>= x.y.z)`) when the package actually breaks below that version.
- Import from where a function is _defined_, not re-exported (e.g. `httr2::req_perform`, not pulled in via another package).

## 5. Function naming and API design

- Use `snake_case` throughout.
- Consider an `object_verb` scheme that mirrors the data model, for example:

```r
observations_get()     # fetch a list of observations
observation_get()      # fetch one by ID
species_list()         # list species in the database
finchdata_auth()       # authenticate / configure credentials
```

- Make the data object the **first argument** so functions are pipe-friendly.
- Functions must **return objects**, never assign into the global environment.
- Keep argument names and order consistent across related functions.

## 6. Documentation (roxygen2)

Every exported function needs `@param`, `@return`, and `@examples`. Enable Markdown in roxygen2 (already done via `Roxygen: list(markdown = TRUE)` in DESCRIPTION).

```r
#' Get bird observations
#'
#' Downloads observation records from the finchdata API for a given
#' geographic bounding box and date range.
#'
#' @param bbox Numeric vector of length 4: `c(xmin, ymin, xmax, ymax)`.
#' @param start_date,end_date Date or character in `"YYYY-MM-DD"` format.
#'   Defaults to the last 30 days.
#' @param limit Integer. Maximum number of records to return. Default `100`.
#'
#' @return A data frame with one row per observation and columns …
#'
#' @examplesIf finchdata_has_auth()
#' obs <- observations_get(
#'   bbox = c(-122.5, 37.7, -122.3, 37.9),
#'   start_date = "2024-01-01"
#' )
#'
#' @export
observations_get <- function(bbox, start_date = NULL, end_date = NULL, limit = 100L) {
  ...
}
```

Additional documentation rules:

- Use `@examplesIf` (with a condition like `finchdata_has_auth()` that checks whether a token is set) rather than `\dontrun{}` for credential-gated examples. pkgcheck flags `\dontrun{}`.
- Add `@family` tags to group related functions.
- Internal helpers get `#' @keywords internal` and `@noRd`.
- `?finchdata` must work — `use_package_doc()` created `R/finchdata-package.R`; fill in the `@description`.
- Run `devtools::run_examples()` locally before pushing.

### README

The README is generated from `README.Rmd` — never edit `README.md` by hand. Required sections, in order:

1. Package name + one-sentence description
2. Badges: CI status, test coverage, [repostatus.org](https://www.repostatus.org/) (start with WIP), and later the rOpenSci review badge
3. Short goal description with links to vignettes
4. Installation instructions (`pak::pak("you/finchdata")`)
5. Extra setup: how to set the API key
6. Brief usage demo (copy-paste runnable with credentials configured)
7. Comparison to similar packages (if any)
8. Citation block

The README must let an editor assess the package without installing it and must assume minimal domain knowledge — define all ornithological/API-specific terms.

### Vignette

At least one HTML vignette is required. Structure it:

1. What the data source is, who issues it, and a link to its documentation
2. Authentication setup
3. Basic usage — a simple query end-to-end
4. Intermediate and advanced patterns

Because the vignette needs credentials, pre-compute it or use `vcr` to record responses. Set `eval = FALSE` where live calls would fail on CRAN, or rely on the vcr cassette.

## 7. Credential handling (security)

This is critical for an API-wrapping package.

```r
# Good: read from environment variable
finchdata_token <- function() {
  key <- Sys.getenv("FINCHDATA_API_KEY", unset = "")
  if (nzchar(key)) return(key)
  # Fall back to keyring if available
  if (requireNamespace("keyring", quietly = TRUE)) {
    return(keyring::key_get("finchdata"))
  }
  cli::cli_abort(
    "No API key found. Set {.envvar FINCHDATA_API_KEY} in your {.file .Renviron}.",
    call = NULL
  )
}
```

Rules:

- Keys come from environment variables or `keyring` — never from a hardcoded default or a function argument that encourages users to type the key inline.
- **Never print the key** — not in messages, errors, or debug output.
- Guide users to `usethis::edit_r_environ()` in the README/vignette so keys never land in committed scripts.

## 8. Tests

```r
# tests/testthat/test-observations.R
describe("observations_get()", {
  it("returns a data frame on success", {
    with_mock_api({                          # httptest2 / vcr cassette
      result <- observations_get(
        bbox = c(-122.5, 37.7, -122.3, 37.9)
      )
      expect_s3_class(result, "data.frame")
      expect_true(nrow(result) > 0)
    })
  })

  it("errors informatively when bbox has wrong length", {
    expect_error(
      observations_get(bbox = c(1, 2)),
      "bbox"
    )
  })

  it("skips live requests on CRAN", {
    skip_on_cran()
    # ...
  })
})
```

Key requirements:

- Cover major functionality **and** error paths — test that things fail the way they should, not just the happy path.
- **Coverage must reach at least 75%** before rOpenSci review; aim higher.
- Use **httptest2** or **vcr** to record/replay API responses so tests run offline and on CI without real credentials. Inspect all recorded fixtures before the first commit to confirm no keys are embedded.
- `skip_on_cran()` for any test that hits the network (or use `skip_if_offline()` which implies it).
- Tests in `test_that()` / `it()` blocks only; no loose code in test files.

## 9. Code style

- Follow the [tidyverse style guide](https://style.tidyverse.org/).
- Format with **Air** (`air format`) or `styler::style_pkg()`.
- Lint with `lintr::lint_package()`.
- Run both before every commit.
- Use `cli::cli_abort()`, `cli::cli_warn()`, `cli::cli_inform()` for all user-facing messages. Never `print()` or `cat()` outside a `print.*()` method.
- Give users a package-level verbosity option (e.g. `options(finchdata.verbose = FALSE)`) rather than per-function arguments.
- Use one consistent input-validation approach across all functions (e.g. always check with `cli::cli_abort()` and `rlang::arg_match()`).
- No startup messages — no version banners on `library(finchdata)`.

## 10. Continuous Integration

Set up GitHub Actions with `usethis`:

```r
usethis::use_github_action("check-standard")   # R-release + oldrel + devel on Linux
usethis::use_github_action("test-coverage")    # Codecov
```

The `check-standard` workflow from `r-lib/actions` tests on **release, oldrel, and devel** R — all three are required by rOpenSci.

Because `finchdata` makes network calls and handles encoding/text, add macOS and Windows to the matrix as well (edit the generated workflow YAML to set `os: [ubuntu-latest, macos-latest, windows-latest]`).

Also add the rOpenSci pkgcheck action so your package is continuously checked against review requirements:

```yaml
# .github/workflows/pkgcheck.yml
# See: https://github.com/ropensci-review-tools/pkgcheck-action
```

Add CI and coverage badges to `README.Rmd` (usethis inserts them when you run `use_github_action()`).

### Required CI matrix summary

| Check         | Platforms                | R versions             |
| ------------- | ------------------------ | ---------------------- |
| R CMD check   | Ubuntu + macOS + Windows | release, oldrel, devel |
| Test coverage | Ubuntu                   | release                |
| pkgcheck      | Ubuntu                   | release                |

## Summary: required vs. recommended

### Required (gates — must have before rOpenSci review)

- Name not on CRAN/Bioconductor; all lowercase
- `Authors@R` with roles and ORCIDs
- CRAN/OSI-accepted license
- `URL` and `BugReports` in DESCRIPTION
- Data-source identification + URL in Description field
- `Imports` not `Depends`
- roxygen2 docs on every exported function (`@param`, `@return`, `@examples`)
- `?finchdata` package-level doc
- README generated from `README.Rmd` with CI and coverage badges
- At least one HTML vignette
- testthat suite; coverage ≥ 75%
- CI on release + oldrel + devel R
- API keys from env vars / keyring; never printed
- `main` as default branch; `NEWS.md` present

### Strongly recommended

- `pak::pkg_name_check()` passing cleanly
- `@examplesIf` instead of `\dontrun{}`
- HTTP fixtures (vcr/httptest2) inspected and scrubbed before first commit
- Multi-OS CI (Ubuntu + macOS + Windows) — required given network/encoding use
- pkgcheck GitHub Action in CI
- 2FA on your GitHub account
- `keyring` support as alternative credential store
- repostatus.org badge in README
