# How to Conduct an rOpenSci Package Review

Congratulations on agreeing to review! Here is a practical walkthrough of the process, followed by the exact checklist template you'll submit.

---

## Before You Start

**Check for conflicts of interest.** If you have (or had) a close working relationship with any of the package authors, tell the editor before starting rather than after.

**Timing.** You agreed to review, so aim to submit within **3 weeks**. Let the editor know early if something comes up.

**Read the submission first.** Open the `ropensci/software-review` GitHub issue for the package and read:

- The submission comment (scope, description, links)
- The README
- The `@ropensci-review-bot` pkgcheck report attached to the issue
- Any CI / Codecov badges

**Optional but recommended: use pkgreviewr.** It clones the repo and scaffolds a review document in RStudio:

```r
# install.packages("pkgreviewr")
pkgreviewr::pkgreview_create(
  pkg_repo = "owner/repo",
  review_parent = "~/reviews"
)
```

---

## Hands-On Checks

Do not review from reading alone — exercise the package:

1. **Install it as documented** and confirm it succeeds.
2. Run `devtools::check()` locally; all notes/warnings/errors matter.
3. Run `devtools::test()` and confirm all tests pass. If any are skipped, check whether the skips are justified.
4. Work through each vignette and the function examples; confirm they run to completion.
5. **Try it on your own data or a real problem.** Climate data packages often wrap an API — actually hit the API, retrieve some data, and tidy it. This is where genuine UX issues surface that automated checks miss.
6. Review the pkgcheck report and CI logs from the issue.

---

## What to Evaluate Beyond the Checkboxes

The checklist covers the basics; good reviews also comment on:

1. **Packaging guideline compliance** — function naming, DESCRIPTION fields, export choices, etc.
2. **Code quality** — tidiness, unnecessary duplication, overly complex logic.
3. **Dependencies** — are all dependencies justified? Could any be replaced with base R or a lighter package?
4. **API design** — do function names and argument names read naturally and autocomplete sensibly? For a climate data package, think about how users will type `get_climate_data(station = ..., date_range = ...)` vs alternatives.
5. **UX** — are error messages informative? Does the package fail gracefully when the API is unreachable or returns unexpected data?
6. **Documentation** — is there a clear statement of need? Do examples show a complete workflow from data retrieval to a tidy result?
7. **Performance** — are there any obvious bottlenecks (e.g., row-by-row API calls that could be batched)?
8. **Real-world robustness** — does it handle missing values, unusual date ranges, or edge cases in API responses?

---

## How to Submit the Review

Post your completed review as a **comment on the GitHub issue**. Use the exact template below (copy-paste it, check boxes, and add your comments). If you open issues or PRs on the package repository itself, link back to them from your review comment.

---

## The Official Review Template

Paste this into your review comment. Check boxes that apply and elaborate in the Review Comments section — the boxes alone are not enough.

```markdown
## Package Review

_Please check off boxes as applicable, and elaborate in comments below. Your review is not limited to these topics, as described in the reviewer guide_

- **Briefly describe any working relationship you have (had) with the package authors.**
- [ ] As the reviewer I confirm that there are no conflicts of interest for me to review this work (if you are unsure whether you are in conflict, please speak to your editor before starting your review).

#### Documentation

The package includes all the following forms of documentation:

- [ ] A statement of need: clearly stating problems the software is designed to solve and its target audience in README
- [ ] Installation instructions: for the development version of package and any non-standard dependencies in README
- [ ] Vignette(s): demonstrating major functionality that runs successfully locally
- [ ] Function Documentation: for all exported functions
- [ ] Examples: (that run successfully locally) for all exported functions
- [ ] Community guidelines: including contribution guidelines in the README or CONTRIBUTING, and DESCRIPTION with URL, BugReports and Maintainer (which may be autogenerated via Authors@R).

#### Functionality

- [ ] Installation: Installation succeeds as documented.
- [ ] Functionality: Any functional claims of the software have been confirmed.
- [ ] Performance: Any performance claims of the software have been confirmed.
- [ ] Automated tests: Unit tests cover essential functions of the package and a reasonable range of inputs and conditions. All tests pass on the local machine.
- [ ] Packaging guidelines: The package conforms to the rOpenSci packaging guidelines.

Estimated hours spent reviewing:

- [ ] Should the author(s) deem it appropriate, I agree to be acknowledged as a package reviewer ("rev" role) in the package DESCRIPTION file.

---

### Review Comments
```

## A Few Tips for Filling It In

- Only check a box if it **genuinely holds**. If something is absent or broken, leave the box unchecked and explain why in the comments — silent unchecked boxes are not helpful.
- Write an honest **hours estimate**; it helps editors understand reviewer workload.
- The `"rev"` DESCRIPTION role is optional and only happens with your explicit consent.
- After you post, the editor (or bot) will record it with `@ropensci-review-bot submit review <url> time <hours>`.

---

## Conduct

Reviews are public and signed. Be specific, constructive, and respectful — the goal is to help the author improve the software, not to gatekeep. If anything gets heated, bring it back to the issue thread or loop in the editor.

The rOpenSci [code of conduct](https://ropensci.org/code-of-conduct/) applies throughout.
