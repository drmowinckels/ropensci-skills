# Setting Up `finchdata` to rOpenSci Standards

This guide walks you through every aspect of setting up `finchdata` — a package that downloads bird observation data from an online API — so it meets rOpenSci's standards from day one.

---

## 1. Package Naming

`finchdata` is a good candidate name: it is lowercase, descriptive, and short. Before you go further, verify it's available:

```r
pak::pkg_name_check("finchdata")
```

This checks CRAN, Bioconductor, and flags potential conflicts. Also do a quick web search to confirm the name isn't offensive in other languages and doesn't conflict with any commercial service's branding.

---

## 2. Create the Package Scaffold

Use `usethis` for all scaffolding — it produces exactly the layout rOpenSci expects:

```r
usethis::create_package("finchdata")
usethis::use_git()
usethis::use_github()         # creates the repo and sets remote
usethis::git_vaccinate()      # populates .gitignore with common cruft
usethis::use_readme_rmd()
usethis::use_vignette("finchdata")
usethis::use_testthat()
usethis::use_package_doc()    # creates R/finchdata-package.R for ?finchdata
usethis::use_citation()       # creates inst/CITATION
usethis::use_news_md()        # creates NEWS.md
```

Make sure the default branch is `main`, not `master`:

```r
usethis::git_default_branch_rename(to = "main")
```

## 3. DESCRIPTION

Open `DESCRIPTION` and fill it in carefully. Here is an annotated template:

```
Package: finchdata
Title: Download Bird Observation Data from the Finch API
Version: 0.0.0.9000
Authors@R:
    person("Your", "Name",
           email = "you@example.com",
           role = c("aut", "cre"),
           comment = c(ORCID = "0000-0000-0000-0000"))
Description: Provides functions to query and download bird observation
    records from the Finch API <https://api.finchdata.example.com>.
    The Finch API is maintained by the Example Ornithology Consortium
    and provides access to global citizen-science bird sightings.
License: MIT + file LICENSE
URL: https://github.com/you/finchdata, https://you.github.io/finchdata
BugReports: https://github.com/you/finchdata/issues
Imports:
    httr2,
    jsonlite,
    cli
Suggests:
    testthat (>= 3.0.0),
    httptest2,
    knitr,
    rmarkdown,
    vcr
Encoding: UTF-8
Roxygen: list(markdown = TRUE)
RoxygenNote: 7.x.x
VignetteBuilder: knitr
```

**Rules to follow:**

- **Title**: Title Case, no trailing period, no "in R" or "with R". Quote external software names with single quotes if needed (e.g., `'eBird'`).
- **Description**: Do not start with the package name or "This package". Wrap URLs in angle brackets `<https://...>`. Since `finchdata` wraps an external data source, include (1) identification of the organization issuing the data and (2) a URL to the access/description page — this is required for data-source packages.
- **Authors@R**: Required form. Assign roles: `aut` (author), `cre` (creator/maintainer), `ctb` (contributor), `cph` (copyright holder). Add your ORCID via `comment = c(ORCID = "...")`. Do not list editors in this field.
- **License**: Use a CRAN- or OSI-accepted license. MIT is a common choice. Mark copyright holders with the `cph` role.
- **URL** and **BugReports**: Both should be present, pointing to your repo and issue tracker.

## 4. Dependencies

For a package that hits an HTTP API and returns JSON, the right dependencies are:

| Task                 | Use                                            |
| -------------------- | ---------------------------------------------- |
| HTTP requests        | `httr2` (preferred over `httr`, `RCurl`)       |
| JSON parsing         | `jsonlite` (preferred over `rjson`, `RJSONIO`) |
| User-facing messages | `cli`                                          |

**Rules:**

- List all runtime dependencies under `Imports`, not `Depends`. `Depends` is almost never correct for modern packages.
- Put `testthat`, `httptest2`, `vcr`, `knitr`, and `rmarkdown` in `Suggests`.
- Avoid specifying minimum versions unless a specific version is actually required for your code to work.
- Import from where functions are defined. If you use `%>%`, import from `magrittr`, not from `dplyr`.
- Do not add a dependency just for convenience if base R covers it adequately.

## 6. Function and Argument Naming

- Use `snake_case` throughout. This is strongly recommended by rOpenSci.
- Consider an `object_verb()` scheme for related functions, e.g.:
  - `finch_search()` — search observations
  - `finch_get()` — retrieve a specific record
  - `finch_species()` — list available species
- Make the primary data object the first argument so functions are pipe-friendly.
- Keep argument names consistent across related functions.
- Functions should return objects, never assign into the global environment.
- Avoid names that clash with base R or popular packages.

## 7. Documentation with roxygen2

Every exported function must have complete roxygen2 documentation. Use Markdown (enabled by `Roxygen: list(markdown = TRUE)` in DESCRIPTION):

```r
#' Search Bird Observations
#'
#' Queries the Finch API for bird observations matching the given criteria.
#'
#' @param species A character string giving the species code (e.g., `"AMRO"`).
#' @param region A character string giving the region code (e.g., `"US-NY"`).
#' @param max_results An integer giving the maximum number of records to return.
#'   Defaults to `100`.
#'
#' @return A data frame with one row per observation and columns for species,
#'   location, date, and count.
#'
#' @examplesIf finch_has_key()
#' finch_search("AMRO", region = "US-NY")
#'
#' @family finch queries
#' @export
finch_search <- function(species, region, max_results = 100) {
  ...
}
```

**Rules:**

- Every exported function needs `@param`, `@return`, and `@examples`.
- Use `@examplesIf` (not `\dontrun{}`) for examples that require API credentials. `pkgcheck` flags bare `\dontrun{}`.
- Internal/unexported functions get `#' @keywords internal` and `@noRd` (no `.Rd` file generated).
- Use `@family` to group related functions.
- Create the package-level doc with `usethis::use_package_doc()` so `?finchdata` works.
- Create a `CITATION` file with `usethis::use_citation()` using `bibentry()` (not the deprecated `citEntry()`). For a package wrapping an external data source, use `citHeader()` to instruct users to also cite the underlying API/database.

## 8. README

The README is generated from `README.Rmd` — never edit `README.md` directly. After editing `README.Rmd`, rebuild with:

```r
devtools::build_readme()
```

Required sections, in order:

1. Package name and one-line description
2. **Badges**: GitHub Actions CI status, Codecov coverage, repostatus.org badge (set to "WIP" while in development, "Active" when ready). Add the rOpenSci peer-review badge after submission.
3. Short goal description with a link to the vignette
4. Installation instructions (via `pak` or `remotes`)
5. Authentication setup (how to get and store an API key — see Security section)
6. A brief, runnable usage demo
7. Brief comparison to similar packages (if any exist)
8. Citation info

The README must let someone assess the package without installing it, and should assume minimal domain knowledge. Since `finchdata` wraps a specific API, explain what that API is and link to its documentation.

## 9. Vignette

At minimum one HTML vignette is required. It should progress from basic to advanced usage:

```r
usethis::use_vignette("finchdata", title = "Getting Started with finchdata")
```

In the vignette for a data-access package:

- Explain what the underlying data source is (don't just say "provides access to the Finch API" — say what the Finch API contains, who maintains it, and what kinds of questions it can answer).
- Link to the API's own documentation.
- Show a realistic workflow: authenticate → search → filter → visualize.
- Cache API responses for the vignette build using `vcr`, or pre-compute and cache the output, so the vignette doesn't require live credentials during `R CMD check`.

## 10. Testing

Set up testthat with:

```r
usethis::use_testthat()
```

**Rules:**

- Write tests as you write each function — tests are part of "done."
- Cover major functionality **and** error paths (test that things fail the right way, not just the happy path).
- Keep tests self-contained; avoid code outside `test_that()` blocks.
- **Coverage must be at or above 75%** to pass rOpenSci review without justification. Aim for higher.

For an HTTP API package, use `httptest2` or `vcr` to record and replay HTTP interactions so tests don't hit the live API:

```r
usethis::use_package("httptest2", type = "Suggests")
# or
usethis::use_package("vcr", type = "Suggests")
```

Example test structure:

```r
# tests/testthat/test-finch-search.R
test_that("finch_search returns a data frame", {
  httptest2::with_mock_api({
    result <- finch_search("AMRO", region = "US-NY")
    expect_s3_class(result, "data.frame")
    expect_named(result, c("species", "location", "date", "count"))
  })
})

test_that("finch_search errors informatively on bad species code", {
  httptest2::with_mock_api({
    expect_error(
      finch_search("NOTABIRD"),
      "species code"
    )
  })
})
```

Skip live API tests on CRAN:

```r
skip_if_offline()   # already implies skip_on_cran() in recent testthat
```

## 11. Security: API Key Handling

Since `finchdata` calls an external API, credential security is required from the start.

**Rules:**

- Read the API key from an environment variable, not a function argument:

```r
finch_key <- function() {
  key <- Sys.getenv("FINCHDATA_KEY")
  if (identical(key, "")) {
    cli::cli_abort(
      "No API key found. Set {.envvar FINCHDATA_KEY} in your {.file .Renviron}.",
      call = NULL
    )
  }
  key
}
```

- Optionally support the `keyring` package as an alternative to `.Renviron`.
- **Never print the key** — not in messages, not in errors, not in verbose output.
- Provide a helper that guides users to store their key safely:

```r
finch_set_key <- function() {
  usethis::edit_r_environ()
}
```

- In your README, document how to get and store an API key without showing real keys.

**Recorded fixtures / test security:**

- When using `httptest2` or `vcr` to record HTTP interactions, inspect every recorded fixture before the first commit. Ensure no keys appear in headers or response bodies.
- Add `inst/fixtures/` or the vcr cassette directory to `.gitignore` during development, then review and selectively commit only scrubbed fixtures.

**Issue templates:**

- Add a note in your GitHub issue template telling users not to share credentials.

## 12. Code Style

- Follow the tidyverse style guide (<https://style.tidyverse.org/>).
- Format code with **Air** (`air format`) or **styler**; lint with **lintr**.
- Use `<-` for assignment; be consistent throughout the package.
- Use `cli` (already in your `Imports`) for all user-facing messages. Never use `print()` or `cat()` outside of `print.*()` methods — they cannot be suppressed.
- Give users a way to turn off verbose output via a package option or environment variable (e.g., `FINCHDATA_QUIET=true`), not just a function argument.
- Use one consistent approach to input validation throughout (e.g., always use `cli::cli_abort()` for errors).
- No startup messages or citation nags when the package loads.

## 13. Continuous Integration

Add GitHub Actions workflows:

```r
usethis::use_github_action("check-standard")   # R CMD check
usethis::use_github_action("test-coverage")    # Codecov
```

The `check-standard` workflow tests against **release, oldrel, and devel** R on Linux, macOS, and Windows — this covers the required R versions. For an API client with no compiled code and no system calls, multi-OS CI is not strictly required, but it is still recommended and it is what `check-standard` provides.

Also wire in the rOpenSci pkgcheck action for continuous compliance checking:

```yaml
# .github/workflows/pkgcheck.yaml
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  pkgcheck:
    uses: ropensci-review-tools/pkgcheck-action/.github/workflows/pkgcheck.yml@main
```

Add CI status and Codecov badges to `README.Rmd`.

For CI secrets: the `FINCHDATA_KEY` secret in GitHub Actions is not available to PRs from forks. Tests that require it should check for the key's presence and skip if it is absent. Document this in `CONTRIBUTING.md`.
