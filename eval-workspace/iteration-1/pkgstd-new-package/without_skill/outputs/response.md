# Setting up `finchdata` to rOpenSci Standards

Here is a complete walkthrough for creating `finchdata` from scratch, following rOpenSci conventions throughout.

---

## 1. Package Creation

Use `usethis` to scaffold the package:

```r
usethis::create_package("finchdata")
```

This creates the canonical R package structure: `R/`, `DESCRIPTION`, `NAMESPACE`, `.Rbuildignore`.

Then immediately initialize git and set up GitHub:

```r
usethis::use_git()
usethis::use_github()
```

---

## 2. DESCRIPTION

The `DESCRIPTION` file is your package's identity document. Fill it carefully.

Key fields:

```
Package: finchdata
Title: Download Bird Observation Data from [API Name]
Version: 0.0.0.9000
Authors@R: 
    person("Your", "Name", , "you@example.com", role = c("aut", "cre"),
           comment = c(ORCID = "0000-0000-0000-0000"))
Description: Provides an interface to the [API Name] API for downloading
    bird observation data including species occurrences, counts, and
    associated metadata.
License: MIT + file LICENSE
Encoding: UTF-8
Roxygen: list(markdown = TRUE)
RoxygenNote: 7.x.x
URL: https://github.com/yourorg/finchdata
BugReports: https://github.com/yourorg/finchdata/issues
```

Notes:
- Use a development version number (`0.0.0.9000`) until first release
- Include your ORCID — rOpenSci encourages this
- `Roxygen: list(markdown = TRUE)` enables Markdown in roxygen2 docs
- `MIT + file LICENSE` is the most common rOpenSci license; run `usethis::use_mit_license()` to create the LICENSE files

## 4. Naming Conventions

- **Package name**: lowercase, no dots, no underscores — `finchdata` is fine
- **Function names**: `snake_case`, verb-first — `fetch_observations()`, `get_species()`, `build_query()`
- **Internal helpers**: prefix with `.` or keep unexported — `parse_response()` as an unexported function
- **File names**: match the primary function or function group they contain — `fetch_observations.R` for `fetch_observations()`
- **Arguments**: `snake_case`, descriptive — `start_date`, `species_code`, `max_results`
- **Constants/configs**: `SCREAMING_SNAKE_CASE` only for true package-level constants

Avoid generic names that conflict with base R or common packages (`print`, `summary`, `data`, `get`).

## 5. HTTP Requests and API Dependencies

For a data-download package, use `httr2` (not the older `httr`):

```r
usethis::use_package("httr2")
```

A minimal request pattern:

```r
fetch_observations <- function(species_code, start_date, end_date, ...) {
  resp <- httr2::request("https://api.example.com/observations") |>
    httr2::req_url_query(
      species = species_code,
      start   = start_date,
      end     = end_date
    ) |>
    httr2::req_perform()
  
  httr2::resp_body_json(resp)
}
```

For authentication (API keys), use `keyring` or advise users to set environment variables. Never hard-code credentials. A standard pattern:

```r
finchdata_key <- function() {
  key <- Sys.getenv("FINCHDATA_API_KEY")
  if (!nzchar(key)) {
    cli::cli_abort(
      "No API key found. Set {.envvar FINCHDATA_API_KEY} in your {.file .Renviron}."
    )
  }
  key
}
```

## 6. User-Facing Messaging

Use `cli` for all messages, warnings, and errors — not `cat()`, `message()`, or `stop()`:

```r
usethis::use_package("cli")
```

```r
cli::cli_inform("Fetching {nrow(results)} records for {.val {species_code}}.")
cli::cli_warn("Rate limit approaching. Pausing before next request.")
cli::cli_abort("API returned status {status}: {message}.")
```

This gives users consistent, styled output and respects `suppressMessages()`.

## 7. Documentation

### Function documentation

Every exported function needs roxygen2 docs:

```r
#' Fetch bird observations
#'
#' Downloads observation records from the API for a given species and date range.
#'
#' @param species_code Character. The species code (e.g., `"AMRO"` for American Robin).
#' @param start_date Date or character in `"YYYY-MM-DD"` format.
#' @param end_date Date or character in `"YYYY-MM-DD"` format.
#' @param max_results Integer. Maximum records to return. Default `1000L`.
#'
#' @return A data frame with columns: `species`, `date`, `location`, `count`.
#'
#' @examplesIf interactive()
#' obs <- fetch_observations("AMRO", "2024-01-01", "2024-06-30")
#'
#' @export
fetch_observations <- function(species_code, start_date, end_date, max_results = 1000L) {
  # ...
}
```

Note `@examplesIf interactive()` — use this for examples that hit a live API, so they are skipped on CRAN.

### Package-level documentation

Create `R/finchdata-package.R`:

```r
#' @keywords internal
"_PACKAGE"

## usethis namespace: start
#' @importFrom lifecycle deprecated
## usethis namespace: end
NULL
```

### README

```r
usethis::use_readme_rmd()
```

Write the README in `README.Rmd` and knit it. It should include:
- One-paragraph description
- Installation instructions (remotes/pak + rOpenSci r-universe)
- Quick usage example
- A link to the pkgdown site

### Vignette

```r
usethis::use_vignette("finchdata")
```

The vignette should walk through a complete real-world workflow: authenticate, query, download, do something with the data.

### pkgdown site

```r
usethis::use_pkgdown_github_pages()
```

This configures a `_pkgdown.yml` and a GitHub Actions workflow to deploy the docs site automatically on push to main.

## 8. Testing

Set up testthat:

```r
usethis::use_testthat()
```

### Test structure (describe/it)

```r
# tests/testthat/test-fetch_observations.R

describe("fetch_observations()", {
  it("returns a data frame with expected columns", {
    withr::with_envvar(c(FINCHDATA_API_KEY = "test-key"), {
      # use httptest2 to mock the API response
      httptest2::with_mock_api({
        result <- fetch_observations("AMRO", "2024-01-01", "2024-01-31")
        expect_s3_class(result, "data.frame")
        expect_named(result, c("species", "date", "location", "count"))
      })
    })
  })

  it("errors with informative message when API key is missing", {
    withr::with_envvar(c(FINCHDATA_API_KEY = ""), {
      expect_error(
        fetch_observations("AMRO", "2024-01-01", "2024-01-31"),
        class = "finchdata_missing_key"
      )
    })
  })
})
```

### Mocking HTTP

For a package that calls an API, **never hit the live API in tests**. Use `httptest2`:

```r
usethis::use_package("httptest2", type = "Suggests")
```

Record real API responses once into `tests/testthat/` fixtures, then play them back. This makes tests fast, reproducible, and CI-friendly without an API key.

### Environment variables in tests

Use `withr::with_envvar()` to set/unset env vars within a test. Add `withr` to Suggests:

```r
usethis::use_package("withr", type = "Suggests")
```

## 10. Continuous Integration

Use r-lib's official GitHub Actions:

```r
usethis::use_github_action("check-standard")
usethis::use_github_action("pkgdown")
usethis::use_github_action("test-coverage")
```

This creates:
- `R-CMD-check.yaml` — runs on Linux/macOS/Windows across R versions
- `pkgdown.yaml` — builds and deploys the docs site
- `test-coverage.yaml` — reports coverage via codecov

Add a codecov badge to your README after setting up the coverage workflow.

For API-dependent tests in CI, store the API key as a GitHub repository secret (`FINCHDATA_API_KEY`) and reference it in the workflow, or ensure `httptest2` mocks make the key unnecessary.

## 11. rOpenSci-Specific Requirements

Before submitting for rOpenSci peer review, you must also:

### Run `pkgcheck`

```r
# install.packages("pkgcheck", repos = "https://ropensci.r-universe.dev")
pkgcheck::pkgcheck()
```

This runs automated checks that rOpenSci editors run before assigning reviewers. Fix all failures before submitting.

### `goodpractice`

```r
goodpractice::gp()
```

Flags style issues, complexity, test coverage gaps, and documentation completeness.

### Minimum test coverage

rOpenSci expects **>= 75% test coverage** (higher is better). Check with `covr::package_coverage()`.

### `CONTRIBUTING.md` and `CODE_OF_CONDUCT.md`

```r
usethis::use_tidy_contributing()
usethis::use_code_of_conduct("your@email.com")
```

rOpenSci requires a code of conduct that links to their community guidelines.

### Lifecycle badges

If any functions are experimental or may change:

```r
usethis::use_package("lifecycle")
usethis::use_lifecycle()
```

Mark unstable functions with `#' @lifecycle experimental` and `lifecycle::signal_experimental()` in the body.

### `r-universe`

After being accepted, rOpenSci will add your package to their r-universe at `ropensci.r-universe.dev`. You do not need to do anything for this — it happens as part of the onboarding process.
